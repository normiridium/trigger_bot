# 🤖 Trigger Admin Bot (Go)

Подробная документация (Wiki): https://github.com/normiridium/trigger_bot/wiki

Telegram-бот с web-админкой триггеров, GPT-интеграцией, музыкальными и медиа-сценариями.

## 🚀 Быстрый старт

### 1) Установить зависимости
```bash
cd /home/faline/trigger_admin_bot
./scripts/install_deps.sh
```

Базовые зависимости:
- `ffmpeg`, `ffprobe` — обработка аудио/видео (конвертация, извлечение дорожек, проверка параметров медиафайлов).
- `webp` (`img2webp`) — подготовка `.webp`-файлов, в том числе для стикеров и превью.
- `yt-dlp` — скачивание медиа по ссылкам (YouTube, Instagram, SoundCloud и др.).
- `nodejs` + `npm` — нужны `yt-dlp` для части YouTube-ссылок и корректной работы зависимых экстракторов.
- `MongoDB` — основное хранилище триггеров, шаблонов, настроек и служебных данных бота.

Дополнительно для превью `.tgs` в админке:
- Установите `lottie_to_webp` или `lottie_to_webp.sh` в `PATH`, чтобы админка могла строить превью анимированных стикеров (`.tgs`).
- Готовые бинарники: https://github.com/ed-asriyan/lottie-converter/releases

### 2) Настроить `.env`
Скопируйте пример и заполните значения:
```bash
cp .env.example .env
```

Минимально обязательные:
- `TELEGRAM_BOT_TOKEN` — токен Telegram-бота от `@BotFather`.
- `MONGO_URI` — строка подключения к MongoDB.
- `OPENAI_API_KEY` — ключ OpenAI API (нужен только для GPT-функций).

Для Spotify:
- `SPOTIPY_CLIENT_ID` — client id Spotify-приложения.
- `SPOTIPY_CLIENT_SECRET` — client secret Spotify-приложения.

### 3) Собрать и запустить
```bash
set -a && source .env && set +a
/usr/local/go/bin/go build -o trigger_admin_bot .
./trigger_admin_bot
```

## 🧩 Что умеет бот

### Триггеры
- Match-типы:
  - `full` — точное совпадение текста.
  - `partial` — срабатывает, если фрагмент встречается внутри сообщения.
  - `regex` — проверка регулярным выражением.
  - `starts` — срабатывает, если сообщение начинается с указанного текста.
  - `ends` — срабатывает, если сообщение заканчивается указанным текстом.
  - `idle` — срабатывает после периода тишины в чате.
  - `new_member` — срабатывает при входе нового участника.
- Режимы триггера (`trigger_mode`):
  Значение `trigger_mode` определяет контекст, где разрешено срабатывание:
  - `all` — в любых сообщениях.
  - `only_replies` — только в ответах (reply) на сообщения.
  - `only_replies_to_any_bot` — только в reply к любому боту.
  - `only_replies_to_combot` — только в reply к `@combot`.
  - `only_replies_to_combot_no_media` — то же, что выше, но с ограничениями по медиа.
  - `never_on_replies` — не срабатывать в reply-сообщениях.
  - `command_reply` — использовать как ответ на команду.
- Режимы админов (`admin_mode`):
  Ограничивает, кто может запускать триггер:
  - `anybody` — любой участник.
  - `admins` — только администраторы.
  - `not_admins` — только не-администраторы.
- `chance` — вероятность срабатывания в процентах (например, 25 = примерно каждое четвёртое подходящее сообщение).
- `pass_through` — если включено, после срабатывания текущего триггера движок продолжает проверять следующие.

### Action-типы
- `send` — отправить текстовый ответ.
- `send_file` — отправить файл/документ.
- `send_gif` — отправить GIF/анимацию.
- `send_sticker` — отправить стикер.
- `delete` — удалить сообщение (обычно исходное или служебное).
- `delete_user_portrait` — удалить сохранённый портрет пользователя.
- `gpt_prompt` — отправить пользовательский текст в GPT по заданному промпту.
- `gpt_image` — сгенерировать изображение через GPT/Images API.
- `search_image` — найти изображение по запросу и отправить результат.
- `spotify_music_audio` — поиск/скачивание/отправка аудио из Spotify-сценария.
- `music_audio` — универсальный музыкальный сценарий (Spotify + Yandex Music).
- `yandex_music_audio` — обработка ссылок и скачивание треков из Яндекс.Музыки.
- `media_link_audio` — скачивание/обработка медиа по ссылкам (YouTube/Instagram/SoundCloud и др.).
- `media_tiktok_download` — скачивание медиа из TikTok.
- `media_x_download` — скачивание медиа из X (Twitter).
- `user_limit_low_warning` — системное предупреждение о лимитах пользователя.

### Web-админка
- URL: `/trigger_bot` — основной вход в панель управления.
- CRUD триггеров — создание, просмотр, редактирование, удаление триггеров.
- CRUD шаблонов — управление шаблонами и заготовками для быстрых настроек.
- Reorder — изменение порядка триггеров (важно, если несколько условий могут совпадать).
- Импорт/экспорт JSON — перенос конфигурации между окружениями и резервное копирование.
- Настройки — изменение параметров работы бота через UI.
- Перезапуск — запуск restart-процедуры без ручного входа на сервер.
- Авторизация в админке — доступ к панели защищён токеном.

## 🎵 Музыка и медиа

### Spotify (`spotify_music_audio`)
- Ищет треки через Spotify API по текстовому запросу.
- Поддерживает прямые ссылки `open.spotify.com/track/...` и `spotify:track:...`.
- Может показывать интерактивный выбор трека в чате.
- После выбора скачивает и отправляет аудио в Telegram.

### Универсальный музыкальный сценарий (`music_audio`)
- Показывает пользователю выбор сервиса: `Spotify` или `Yandex Music`.
- Работает как с текстовым запросом, так и с прямыми ссылками сервисов.

### Yandex Music (`yandex_music_audio`)
- Принимает ссылки вида `music.yandex.ru/.../track/...`.
- Использует встроенный загрузчик, без отдельного внешнего CLI.

### Медиа по ссылкам (`media_link_audio`)
- Поддерживает YouTube, Instagram, SoundCloud и другие источники, обрабатываемые через `yt-dlp`.
- Для YouTube может показывать выбор: отправить `аудио` или `видео` (если включён интерактивный режим).
- Для Instagram автоматически определяет тип вложения (фото/видео).
- Для SoundCloud сразу запускает аудиосценарий.
- Применяет ограничения по размеру/высоте и при необходимости автоматически транскодирует видео под лимиты Telegram.

### TikTok / X
- `media_tiktok_download` — скачивание и отправка медиа из TikTok-ссылок.
- `media_x_download` — скачивание и отправка медиа из X (Twitter)-ссылок.

## 🗂️ Правила удаления исходного сообщения

Флаг триггера:
- `delete_source_message` — если включён, бот пытается удалить исходное пользовательское сообщение после обработки триггера.

Текущее поведение:
- source-сообщение удаляется **только после успешной отправки результата**.
- При ошибках скачивания/отправки source остаётся в чате.
- При `Отменить` в интерактивных сценариях source не удаляется.

## 🧰 Команды бота
- `/start` — коротко о боте и доступных возможностях.
- `/help` — справка по командам и подсказки по использованию.
- `/emojiid` — показывает ID кастомного emoji.
  Использование: отправьте команду или кастомный emoji (лучше в личку боту).
- `/stickerid` — показывает file ID стикера из сообщения.
  Использование: обычно в reply на сообщение со стикером.
- `/gifid` — показывает ID GIF/анимации и подпись (если есть).
  Использование: в reply на сообщение с GIF.
- `/q [N]` — делает quote-стикер из реплая.
  Использование: `/q` или `/q 3` (взять несколько последних сообщений).
- `/qdel` — удаляет стикер из стикерпака бота.
  Использование: в reply на сообщение со стикером, который нужно удалить.
- `/spsearch <запрос>` — поиск трека в Spotify.
  Пример: `/spsearch daft punk harder better faster stronger`.
- `/summary` — делает краткую сводку по переписке (если функция включена).
- `/myportrait` — показывает ваш сохранённый портрет/персону для GPT-сценариев.
- `/delmyportrait` — удаляет ваш сохранённый портрет/персону.

### Админские команды (модерация)
- `/ban <@user|id|reply> [30m|2h|1d|1w] [причина]` — банит пользователя в чате (срок опционален, без срока — бессрочно).
- `/unban <@user|id|reply> [причина]` — снимает бан и разрешает снова входить/писать.
- `/mute <@user|id|reply> [30m|2h|1d|1w] [причина]` — запрещает пользователю писать сообщения (срок опционален).
- `/unmute <@user|id|reply> [причина]` — снимает мут.
- `/kick <@user|id|reply> [причина]` — удаляет пользователя из чата без длительного ограничения (может зайти снова по ссылке/инвайту).
- `/readonly [30m|2h|1d|1w]` (алиасы: `/ro`, `/channelmode`) — включает режим “только чтение” для всего чата (писать нельзя всем участникам, кроме админов).
- `/reload_admins` — обновляет кеш администраторов чата (полезно после смены состава админов).

Разница между `kick`, `mute` и `readonly`:
- `kick` — действие на одного пользователя: удалить из чата сейчас.
- `mute` — действие на одного пользователя: оставить в чате, но запретить писать.
- `readonly` — действие на весь чат: временно закрыть отправку сообщений для всех обычных участников.

Поддерживаются алиасы с `!`: `!ban`, `!unban`, `!mute`, `!unmute`, `!kick`, `!readonly`, `!reload_admins`.

Тихие варианты (без уведомления): `!sban`/`/sban`, `!sunban`/`/sunban`, `!smute`/`/smute`, `!skick`/`/skick`.

## 📤 Импорт/экспорт

- Доступно в web-админке в разделе `Импорт/экспорт`.
- `Экспорт` — выгружает текущие триггеры и шаблоны в JSON-файл (для бэкапа или переноса).
- `Импорт` — загружает JSON-файл обратно в систему (например, после переезда или восстановления).
- Рекомендуется перед импортом сделать экспорт текущего состояния как резервную копию.
- Файл образца для импорта: `examples/import_starter_anonymized.json`.

Формат импорта — bundle JSON c секциями:
- `triggers` — список правил с условиями и действиями.
- `templates` — набор шаблонов, используемых в админке для быстрого создания триггеров.

## 🔁 Жизненный цикл сообщения

Ниже упрощённая схема, как бот обрабатывает входящее сообщение или событие:

1. Входящее событие.
Бот получает update из Telegram: текст, reply, медиа, сервисное событие (например, новый участник), callback кнопки и т.д.

2. Базовые проверки.
Сообщение проходит первичную валидацию: есть ли нужные поля, не бот ли отправитель, разрешён ли чат (`ALLOWED_CHAT_IDS`), доступен ли контекст для обработки.

3. Индексация и служебный контекст.
Бот обновляет внутренний контекст (например, индекс пользователей, кеш админов, историю для quote/summary), чтобы следующие шаги работали корректно.

4. Проверка на модерационные команды.
Если сообщение похоже на команду модерации (`/ban`, `!mute`, `/readonly` и т.д.), выполняется модерационный обработчик.
При успешной обработке мод-команды обычный триггерный пайплайн для этого сообщения завершается.

5. Подготовка runtime-данных.
Перед матчингом бот использует предподготовленные данные:
- список триггеров берётся из кеша (`ListTriggersCached`), а не перечитывается из БД на каждый update;
- шаблоны ответов парсятся и кешируются в памяти (`responseTemplateCache`);
- regex-триггеры компилируются с кешированием в памяти (`regexGlobalCache`), то есть один и тот же паттерн не компилится заново на каждом сообщении.

6. Фильтрация кандидатов-триггеров.
Из набора активных триггеров выбираются те, что потенциально подходят по контексту:
- кто отправил (`admin_mode`);
- где отправил (`trigger_mode`);
- тип события (обычный текст, reply, new_member, idle и т.д.);
- включён ли триггер.

7. Match-проверка.
Для каждого кандидата проверяется условие совпадения (`full`, `partial`, `regex`, `starts`, `ends`, `new_member`, `idle`).
Если условие не выполнено, бот переходит к следующему триггеру.

8. Вероятностный фильтр (`chance`).
Если у триггера задана вероятность, применяется случайная проверка.
Только прошедшие её триггеры переходят к выполнению action.

9. Выполнение action.
Бот запускает действие триггера (`send`, `gpt_prompt`, `music_audio`, `media_link_audio`, `delete` и т.д.).
Для интерактивных сценариев могут создаваться промежуточные шаги (кнопки выбора, уточнения).

10. Пост-обработка.
Если на фронте включён переключатель `Удалить исход. сообщение`, исходное сообщение удаляется только после успешного завершения действия.
При ошибках или отмене в интерактивном сценарии исходное сообщение остаётся.

11. Остановка или продолжение цепочки.
Если на фронте `Сквозная реакция` выключена, после первого успешного срабатывания обработка обычно останавливается.
Если `Сквозная реакция` включена, бот продолжает проверять и выполнять следующие подходящие триггеры.
